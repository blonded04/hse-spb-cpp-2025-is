# Синтаксис (`0x`)
## `02-trivially-copyable`
Допишите функции `write_to` и `read_from`, которые сохраняют/читают trivially copyable структуру в/из файла.
Будьте осторожны: файл следует открывать в бинарном режиме, иначе будет несовместимость между Windows и Linux/macOS.

Должен получиться файл `02.bin`, наверняка совместимый с `02.mingw-w64-x86_64-gcc.bin`.
Байты padding могут иметь любое значение.
Поймите, как сохранились `int`'ы; это не единственный способ, но популярный (little-endian).

Должен получиться файл `03.bin`, наверняка совместимый с `03.mingw-w64-x86_64-gcc.bin`.

# Для домашки (`1x`)
## `10-siof`
Измените файл `storage.cpp` так, чтобы программа независимо от способа компиляции компилировалась, не падала и выдавала одно из двух: либо
```
10
20
30
100
200
300
```
либо
```
100
200
300
10
20
30
```

Сейчас проблема в том, что единицы трансляции `a.cpp` и `z.cpp` в конструкторах своих глобальных переменных
вызывают функцию `get_numbers()` из единицы трансляции `storage.cpp`, которая возвращает ссылку на
глобальную переменую `numbers`.
Однако эта переменная могла быть ещё не проинициализирована.

**Важно**: `std::cout`, `std::cin`, `std::cerr` - тоже лишь глобальные переменные, отчего они тоже подвержены static initialization order fiasco. Бороться с siof для стандартных потоков вам поможет `std::ios_base::Init`.

# Продвинутый синтаксис (`2x`)
## `20-read-object-as-bytes`
Программа считывает из файла массив размера 50 и трактует некоторые его куски как объекты типа `Point`.
Строго говоря, [до C++20](https://www.open-std.org/jtc1/sc22/wg21/docs/papers/2020/p0593r6.html#objects-provided-as-byte-representation) это UB.
Исправьте UB: перед чтением создайте в массиве объекты `Point`, а в конце программы удалите.

Обсудите с преподавателем, всё ли правильно получилось.
К сожалению, тут UB довольно тонкое/формальное и инструментами не ловится.

Скорее всего, ассемблерный код получился абсолютно одинаковый: компилятор понял,
что конструктор и деструктор `Point` ничего не делают.
Зато нет UB даже в C++17.
