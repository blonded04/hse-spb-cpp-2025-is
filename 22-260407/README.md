# Общее
Цель занятия: проникнуться тем, откуда могут вылетать исключения.

Перед запуском упражнения убедитесь, что вы понимаете, почему текущее решение неверное (даже если компилируется).

Разрешается изменять только код, помеченный `TODO`.
**Запрещается использовать try-block для обеспечения гарантий исключений**,
можно его использовать только для финальной обработки.

Рекомендуется обсудить с преподавателем: `14-add-noexcept`, `21-vector-throws`, `30-copy-no-basic-guarantee`.

# Для домашки (`1x`)
## `10-reading-no-strong`
Реализована функция `read_vector_to`, которая считывает вектор из потока и записывает его в output-аргумент.
Как демонстрирует `main`, эта функция не даёт строгой гарантии исключений: если вылетело исключение, вектор может остаться в непонятном состоянии.
Исправьте это без try-block.

## `11-errno-to-exceptions`
В этой программе каждая функция возвращает `true`, если ошибок не произошло, и `false` иначе.
Поэтому приходится при каждом вызове функции разбираться, не произошло ли ошибки.
Перепишите код на использование исключений, можно использовать стандартный `std::ios_base::failure`.
Теперь все функции возвращают `void` и можно избавиться от output-параметров.
Это заодно обеспечит строгую гарантию исключений.

## `12-raii`
`int_reader` — обёртка над `ifstream` и сейчас бывает два состояния после вызова конструктора: удалось открыть и не удалось,
надо отдельно проверять вызовом `is_open`.
Добавьте более строго концепцию RAII (Resource Acquisition Is Initialization): если конструктору не удалось открыть файл,
выбрасывается исключение.
Можно переиспользовать `read_error`.

Теперь любой `int_reader` в не-moved-from state гарантированно имеет открытый файл.

## `13-please-use-vector`
Зацените количество кода, которое нужно, чтобы вручную написать класс `matrix` с правилом пяти и ручным управлением памятью.
Перепишите только `matrix` с использованием `vector<int>`.
Убедитесь, что получилось сильно короче.
Можно было бы сделать и с `vector<vector<int>>`, но тогда было бы не совсем честное сравнение.

## `14-add-noexcept`
Вам даны классы `Point` и `Points`.
Вообще они следуют правилу нуля, но в этом упражнении все пять специальных методов выписаны явно.
Допишите ко всем методам (включая специальные) `noexcept`, если соответствующий метод не должен и не будет кидать исключение.
Обсудите с преподавателем.

# Продвинутый синтаксис (`2x`)
## `20-raw-resource`
Убедитесь, что в функции `read_print()` при возникновении исключения утекает память (а при корректном вводе не возникает),
то есть она не даёт базовую гарантию исключений.
Исправьте функцию `read_print()`, не меняя семантику.

## `21-vector-throws`
При любом выделении памяти может вылететь `std::bad_alloc`.
Например, при очень многих операциях с `std::vector`.
Поймите, почему две функции не предоставляют базовую гарантию исключений, обсудите с преподавателем.
Исправьте.

Обратите внимание, что исправлению вообще неважно, откуда вылетают исключения.

# Хитрые упражнения (`3x`)
## `30-copy-no-basic-guarantee`
Оказывается, даже если класс следует правилу нуля, но его поля связаны инвариантом,
то некоторые операции с ним могут не обеспечивать даже базовую гарантию.

Напишите в `main` пример кода, в котором чисто теоретически может произойти нарушение
базовой гарантии и падение программы.
Получить его на практике довольно сложно (но если вы под линуксом, можете попробовать поиграть с `ulimit -v` в отдельном шелле).
